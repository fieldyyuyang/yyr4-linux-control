# Device Discovery and Input Adapter

This document details the safety, architecture, and constraints of the YYR4 device discovery and event collection systems.

## Why VID/PID is Not Enough
Relying solely on `VID` (239A) and `PID` (80F4) is unsafe because USB identifiers can be cloned or shared. To prevent accidentally grabbing a different macro pad or malicious device, we enforce strict identity matching.

## Strict Identity Matching
To validate a YYR4 device, we require:
- USB bus type.
- Exact VID (239A) and PID (80F4).
- Manufacturer name containing `YOUYOU Keyb_V2`.
- Product name containing `YOUYOU TEC.`.
- Exactly Interface 02.
- A keyboard and a mouse event node, correctly named.

## Keyboard and Mouse Parent Association
The keyboard and mouse evdev endpoints MUST share the exact same physical USB parent device path (`parent_usb_syspath`). This ensures we don't accidentally mix endpoints from two physically distinct devices plugged into different ports.

## Device Node vs Stable Path
- `device_node` (e.g. `/dev/input/event3`) is a dynamic, volatile kernel assignment. It may change between reboots or replugs.
- `stable_path` (e.g. `/dev/input/by-id/xxx`) is generated by udev and remains consistent. We resolve the stable path for configuration persistence but use the dynamic node strictly for runtime `open()`.

## No Full Serial Preservation
We DO NOT persist or output the raw USB serial number. The identity model only tracks a boolean `serial_present` and a non-sensitive `local_identity_hint` to uniquely identify the device within the current session without logging PII or violating the Vibe Coding security rules.

## Fail-Closed on Multiple Devices
If the system detects multiple valid YYR4 devices, or multiple ambiguous interfaces belonging to the same device, the discovery process will "fail closed". It refuses to blindly guess which one to use, throwing a `DeviceAmbiguousError`.

## Main Keyboard Protection
By strictly adhering to the vendor, product, and interface 02 topology, the discovery process naturally isolates itself from the user's main keyboard and mouse. We will never accidentally read them.

## Current Stage Restrictions (Milestone 1.1)
- **No EVIOCGRAB**: We open the device purely read-only (`EvdevInputAdapter`) without taking exclusive control. This is for safe observation and parsing validation.
- **No uinput**: We do not yet create virtual outputs.
- **No daemon**: Everything runs in isolated processes for testing.

## Optional Dependencies
The core parsing logic remains completely pure Python and dependency-free.
`evdev` and `pyudev` are specified as optional dependencies under `[project.optional-dependencies] linux-input`. 
They are lazily imported. If missing, the backend throws a `DependencyUnavailableError` instead of a hard crash.

## Mock Testing Strategy
All units are tested using injected `DiscoveryBackend` and `EventDeviceFactory` fakes (`FakeDiscoveryBackend`, `FakeEventDeviceFactory`). This ensures 100% test coverage without requiring the `linux-input` dependencies or a real connected device on the testing host.

## Note on Transport Codes
**The user does NOT need to switch transport configurations at this stage.** The current YYR4 device can remain in its everyday configuration. We are only developing the safe discovery layer. Later, during the actual integration tests, the "after profile" will be loaded temporarily.
